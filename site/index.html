Config-Driven Agentic AI：讓 AI Agent 用對話產出零售分析配置

🎤 追客 Fredrick（Data / AI Engineer, Tagtoo）

✨ 從『Workflow Agent』的失敗經驗，提煉出『Config-Driven』新架構模式，實戰教訓具體可借鏡。

本講題分享零售數據分析的實戰開發經驗，針對「高度客製化分析需求與規模化系統之間的拉扯」提出 Config-Driven Agentic AI 架構。傳統 V1 架構讓 Agent 作為調度者依據工作流 YAML 直接執行分析，卻面臨速度慢、Token 成本高與上下文腐敗等不穩定問題。為此，Tagtoo 轉向 V2 架構：讓 AI Agent 僅負責與使用者對話並產出結構化的「分析合約（`project.yaml`）」，內含分析單位（Entity）與篩選邊界（Filter），再交由確定性的 Pipeline 腳本執行。系統依需求頻率設計 L1 腳本、L2 說明書與 L3 自訂 SQL 三種路徑，並建立三道防線（需求分類、合約合法性與 dry-run 驗證、結果需求覆蓋率審查），成功以配置化將業務邏輯與自動化分析流程穩定銜接。

AI Enterprise · #30

零資料接觸架構：讓 LLM 讀懂企業資料但永遠摸不到它

🎤 Chester（Plusblocks Technology）

✨ 『零資料接觸架構（ZDTA）』以 Schema-Only 查詢代理解決 LLM 資料外洩兩難，設計巧妙。

本演講針對企業在導入大語言模型時面臨「既想利用內部資料進行自然語言分析，又害怕資料外洩與法規合規風險」的核心矛盾，提出創新且安全的「零資料接觸架構（Zero Data Touch Architecture, ZDTA）」。講者 Chester 拆解了 ZDTA 的三層安全防禦設計：第一，Schema-Only Context，僅將資料庫 Schema、欄位型別等結構資訊提供給 LLM，不包含任何實際資料；第二，Query Proxy 中間層，利用 AST（抽象語法樹）解析進行 SQL 白名單驗證與欄位級權限控管，防止 LLM 受 Prompt Injection 影響產生惡意 SQL；第三，Result Sanitizer，將資料庫執行的查詢結果進行 PII 遮罩與 K-Anonymity 驗證後直接呈現給用戶，繞過 LLM。實務上透過 Few-shot 範例可將準確度拉升至 88%，並輔以 Schema/Query Plan 快取將額外延遲降至 200ms，為金融、醫療等高監管產業提供了結構化資料安全分析的絕佳實踐藍圖。
