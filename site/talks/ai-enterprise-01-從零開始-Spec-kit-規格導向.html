從零開始 Spec-kit 規格導向 (Workshop)

🎤 Che-Chia Chang　·　SRE @ Maicoin / Microsoft MVP

Spec-driven DevelopmentSpec-kitAI Agent開發流程Workshop

摘要

本工作坊介紹如何使用 Spec-kit 實踐 Spec-driven Development（SDD），透過撰寫可執行規格、範本與自動化腳本，解決團隊導入 AI 寫程式後常見的「Spec drift」問題。講者以 YouBike 即時站點查詢器為範例，帶領學員走過 specify → plan → tasks → implement 的完整流程，並分享何時 SDD 優於 Vibe Coding、如何選擇模型與避免常見失敗模式。

重點整理

  • Spec drift 是多數團隊導入 AI 開發後常見痛點：需求、規格、實作與測試不同步
  • 核心流程為 /specify → /plan → /tasks → /implement，並以 Constitution 定義 agent 行為準則
  • Spec-kit 對模型能力要求不高，用 mini/nano 模型常可達約 90% 效果且成本僅 10–20%
  • 常見失敗模式包括：未寫 Spec 就實作、規格改了但 tasks 未同步更新、規格太模糊
  • 建議把測試覆蓋率、lint、style 等驗收標準直接寫入 Spec，讓 agent 自動驗收

背景與挑戰

多數團隊在導入 AI 協助寫程式後，容易發生需求、規格、實作與測試互相不同步的「Spec drift」現象，導致返工與品質不穩定。本工作坊聚焦如何用 Spec-kit 與 Spec-driven Development（SDD）建立「規格先行、可執行驗證、持續對齊」的開發流程，讓 AI 與工程團隊使用同一份契約協作。

實作演練

講者以 YouBike 2.0 即時站點查詢器為 LAB 範例，示範透過 specify init 初始化專案、撰寫 Constitution 定義 agent 行為準則，再依序執行 /speckit.specify、/speckit.clarify、/speckit.plan、/speckit.tasks、/speckit.analyze 與 /speckit.implement，完整走過從規格到程式碼的流程。

技術重點

Spec-kit 是把 Spec-driven Development 落地成可重複 workflow 的工具，透過 script、template、checklist 讓 agent 可預測地產生規格，並在每一步設置 checkpoint 才能進入下一步。相較於 Vibe Coding，Spec 作為中間產物可作為 source of truth，有利於 review、拆分任務與多人協作。

案例與心得

講者分享 Spec-kit 核心流程對模型能力要求不高，用較便宜的 mini/nano 模型即可達到約九成效果，建議以強模型撰寫 Spec、以 mini 模型執行 implement 來控制成本。同時列舉常見失敗模式，例如尚未寫 Spec 就直接進入實作、規格修改後未同步更新 tasks，以及規格過於模糊導致 agent 自由發揮。

結論與啟示

講者總結三個可帶走的重點：spec → plan → tasks → implement 是可落地的流程；規格優先能降低返工成本；AI agent 要穩定運作，必須搭配 Spec 與驗收標準形成的迴圈。
