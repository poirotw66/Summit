===== PAGE 7 =====
核心理念
LLM 只看結構，不看內容
Schema-Only
Context
只給LLM 資料表結構、欄位名稱
與型別，不給任何一筆資料
Query
Proxy
LLM 產生查詢語句，由安全中間
層代為執行，結果不回傳LLM
Result
Sanitizer
查詢結果經脫敏、聚合後才呈現
給使用者，LLM 全程不經手

===== PAGE 9 =====
Layer 1：Schema-Only Context
LLM 收到的Context
CREATE TABLE orders (
id        INT PK,
customer  VARCHAR(100),
amount    DECIMAL(10,2),
status    ENUM('pending',
'shipped','done'),
created   TIMESTAMP
);
LLM 看不到的
| id | customer    | amount |
|----|-------------|--------|
| 1  | 台積電
| 50M    |
| 2  | 鴻海
| 32M    |
| 3  | 聯發科
| 28M    |
任何一筆真實資料

===== PAGE 10 =====
Layer 2：Query Proxy
LLM 產生SQL → Proxy 驗證→ 安全執行
SQL 白名單驗證
僅允許SELECT，禁止INSERT / UPDATE / DELETE / DROP
欄位級權限控制
依使用者角色限制可查詢的欄位（如：業務看不到成本欄位）
查詢複雜度限制
限制JOIN 深度、子查詢層數、結果筆數上限
Injection 防護
AST 解析而非字串匹配，阻擋LLM 被誘導產生的惡意查詢

===== PAGE 11 =====
Layer 3：Result Sanitizer
脫敏管線
1
PII 偵測與遮罩（姓名、身分證、電話）
2
數值聚合（個別金額→ 統計摘要）
3
K-Anonymity 驗證（≥ 5 筆才回傳）
4
Audit Log 寫入（誰查了什麼）
關鍵設計
查詢結果直接從Sanitizer 送到前端，完全
繞過LLM。LLM 負責「理解意圖」和「產
生查詢」，但永遠不經手任何一筆查詢結
果。

===== PAGE 18 =====
踩過的坑
Schema 太大，Context 爆了
→ 動態Schema 選取：依使用者問題只注入相關表格
LLM 幻覺產生不存在的欄位
→ Proxy 的AST 驗證會攔截，但需要好的錯誤訊息引導retry
聚合過度，結果失去意義
→ 建立「最小脫敏規則」，依資料敏感等級分層處理
延遲增加~800ms
→ Schema cache + Query plan cache，降至~200ms 額外延遲

===== PAGE 19 =====
何時該用ZDTA？
適合場景
•
結構化資料（SQL 資料庫）
•
高度監管行業（金融、醫療）
•
查詢型需求為主
•
已有完善的Schema 文件
•
需要完整Audit trail
不太適合
•
非結構化資料（文件、圖片）
•
需要LLM 做深度推理
•
資料量小且不敏感
•
沒有明確Schema
•
需要即時對話式分析

===== PAGE 20 =====
Key Takeaways
LLM 不需要看到資料，也能理解資料— Schema 就是語言
安全邊界要在架構層實現，而非靠Prompt 約束
準確度的損失可以用Few-shot 和動態Schema 補回來
每一筆查詢都要有Audit trail — 合規的基石
