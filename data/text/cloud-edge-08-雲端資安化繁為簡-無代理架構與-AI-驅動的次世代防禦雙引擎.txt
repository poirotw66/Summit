當 CVE 出現的時候該怎麼處理?
To remediate the Remote Code Execution vulnerability in Apache Tomcat 
(CVE-2025-24813) on your AWS VM, follow these steps:
Step 1: Preparation (準備)
1.
Backup the current state: Ensure you have a backup of the current VM state or 
a snapshot to allow rollback if needed.
2.
Notify stakeholders: Inform relevant teams about the planned changes and 
potential impact.
Step 2: Upgrade Apache Tomcat (升級)
1.
Connect to the VM:
●
Use SSH to connect to the VM using its public IP address.
ssh ec2-user@52.207.56.248
2.
Check the current version of Tomcat:
sudo systemctl status tomcat
3.
Stop the Tomcat service:
sudo systemctl stop tomcat
4.
Download the latest version of Tomcat:
●
Choose the appropriate version based on the recommendation (11.0.3+, 
10.1.35+, or 9.0.99+).
wget 
https://downloads.apache.org/tomcat/tomcat-9/v9.0.99/bin/apache-to
mcat-9.0.99.tar.gz
5.
Extract the downloaded ﬁle:
tar -xvzf apache-tomcat-9.0.99.tar.gz
6.
Backup the existing Tomcat installation:
sudo mv /usr/local/tomcat /usr/local/tomcat-backup
7.
Move the new Tomcat ﬁles to the appropriate directory:
sudo mv apache-tomcat-9.0.99 /usr/local/tomcat
8.
Copy conﬁguration ﬁles from the backup:
Ensure you copy necessary conﬁguration ﬁles from the backup to the new 
installation.
sudo cp /usr/local/tomcat-backup/conf/* /usr/local/tomcat/conf/
9.
Start the Tomcat service:
sudo systemctl start tomcat
10.
Verify the Tomcat service is running:
sudo systemctl status tomcat
Step 3: Validation (驗證)
1.
Check the Tomcat version:
Access the Tomcat manager or use the command line to verify the version.
/usr/local/tomcat/bin/version.sh
2.
Test the application:
Ensure the application is running as expected and there are no disruptions.

當 CVE 出現的時候該怎麼處理?
Step 4: Rollback Procedure (倒回)
1.
If issues are encountered:
Stop the Tomcat service.
sudo systemctl stop tomcat
2.
Restore the backup.
sudo rm -rf /usr/local/tomcat
sudo mv /usr/local/tomcat-backup /usr/local/tomcat
3.
Start the Tomcat service.
sudo systemctl start tomcat
Step 5: Mitigation (if upgrade is not possible immediately) (替代⽅案)
1.
Modify the servlet conﬁguration:
Edit the web.xml ﬁle to set readonly='true'.
sudo vi /usr/local/tomcat/conf/web.xml
2.
Disable partial PUT support:
Ensure the readonly attribute is set to true for the DefaultServlet.
<servlet>
<servlet-name>default</servlet-name>
<servlet-class>org.apache.catalina.servlets.DefaultServlet</servlet-class>
<init-param><param-name>readonly</param-name>
<param-value>true</param-value>
</init-param>
</servlet>
3.
Ensure security-sensitive ﬁles are not in public upload paths:
Review and adjust ﬁle storage locations as necessary.
Step 6: Compliance and Documentation (合規⽂件)
1.
Document the changes: Record all changes made, including the version upgrade and 
conﬁguration modiﬁcations.
2.
Review compliance: Ensure the changes comply with relevant security and regulatory 
requirements. By following these steps, you can remediate the vulnerability while adhering 
to security best practices and ensuring minimal disruption to production workloads.
